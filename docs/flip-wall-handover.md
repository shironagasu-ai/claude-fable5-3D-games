# Flip Wall 引き継ぎ資料（開発ハンドブック）

対象: `games/flip-wall/index.html`（単一ファイル・約900行）
目的: 別モデル / 別セッションで開発を続けるために、設計思想・不変条件・既知の落とし穴・検証手順をまとめる。

---

## 1. アプリ概要と確定済みの仕様

ユーザーがアップロードした AI イラスト（最大6枚）を、数千〜20万個の立方体（ボクセル）で
構成された壁に表示するインタラクティブ・ギャラリー。立方体の6面 = 6枚の画像。

ユーザーと合意済みの仕様（変えるときは要相談）:

- **壁クリック** = クリック地点を波源に「次の絵」へ切り替え
- **サムネイル / 数字キー1-6** = その絵へ中央から切り替え。表示中の絵を再選択するとスピン波
- **画像1枚だけでも動作**。1枚のときの壁クリックはスピン波（同じ絵に戻る）
- 未アップロード時は内蔵プロシージャル画像6枚（数字入り）でデモ動作
- 巡回対象: ユーザー画像が1枚でもあれば**ユーザー画像のみ**、無ければ内蔵6枚
- 切り替えパターン5種＋おまかせ: 波紋 / スイープ / 渦 / ランダム / 集合（`PATTERNS` 配列）
- 📷 シネマカメラ（Cキー）: 6ショットをランダム順カット割り。シネマ中は約9秒ごと自動切替
- 🏁 FLIP MARK ベンチマーク: プリセットごとに決定的シーケンスで 平均FPS / 1% Low / 切替コストを計測
- 👁 UI非表示モード（Hキー）: 鑑賞・スクショ用。非表示中も壁操作は有効
- 粒子数プリセット: 5K / 20K / 50K / 100K / 200K（モバイル初期値 2,200）
- **画像は完全ブラウザ内処理**（送信・保存なし）。localStorage に入れるのはベンチスコアのみ
- **効果音なし**（ユーザー決定で WebAudio 処理を全カット済み。再追加しないこと）

## 2. リポジトリ全体の方針（全ゲーム共通）

- ビルド工程なし。GitHub Pages（main ブランチのルート）でそのまま動く静的ファイルのみ
- 外部 CDN 禁止。ライブラリは `vendor/` にバージョン固定で同梱（three.js 0.170.0 ほか）
- 画像・音声などのアセットファイル不使用（テクスチャは Canvas 生成、音は WebAudio 合成）
- 1ゲーム = `games/<name>/index.html` の自己完結1ファイル
- 追加・変更はヘッドレスブラウザの E2E テストで通し検証してからコミット（→ §6）
- コミットメッセージは英語・機能単位でまとめる。開発ブランチ:
  `claude/fable-3d-games-exploration-yqr0zy` → PR で main へ

## 3. アーキテクチャ（GPU駆動アニメーション）

### 描画
- 全キューブを **1つの `InstancedBufferGeometry` + `ShaderMaterial` + 1ドローコール**で描画
  （`InstancedMesh` ではなく素の `THREE.Mesh`。`frustumCulled = false` 必須）
- **毎フレームの CPU 仕事は `uNow`（経過秒）ユニフォームの更新のみ**。
  行列転送ゼロ。これが 20万粒子を成立させている核心。壊さないこと

### インスタンス属性（切り替え時のみ一括書き込み）
| 属性 | 型 | 意味 |
|---|---|---|
| `aGrid` | vec2 | グリッド座標（テクスチャUV兼ワールド位置の元）。再構築時のみ設定 |
| `aStart` | float | アニメ開始絶対時刻（`clock.elapsedTime` 基準） |
| `aDur` | float | 持続秒。**0 なら「静止＝aQ1 の姿勢」**（初期状態はこれ） |
| `aKind` | float | 0=flip / 1=spin / 2=assemble |
| `aScatter` | vec3 | 集合パターンの飛散ベクトル |
| `aQ0` / `aQ1` | vec4 | 回転の始点 / 終点クォータニオン |

- 頂点シェーダー内で `u = clamp((uNow-aStart)/aDur, 0, 1)` → 種別ごとに
  nlerp（符号合わせ必須: `q1 * sign(dot(q0,q1))`）/ Y軸回転合成 / 飛散オフセットを計算
- flip のイージング `easeBack` は u=1 で厳密に 1 になるので終端誤差なし

### JS 側ミラー（jsQ0 / jsQ1 / jsStart / jsDur / jsKind）
- GPU 属性と同内容を JS の TypedArray に保持
- 用途: **アニメ途中の再トリガー**時に `currentQuatOf()` で現在姿勢を計算して
  次の `aQ0` にする（これが無いと連打でワープする）。GLSL と同じ式を JS に持つこと
- `animatingCount()`（テスト用）もこのミラーから計算

### トリガーの流れ（`triggerWave(target, ox, oy, mode, rng)`）
1. パターン決定（`mode` 優先 / 'auto' はランダム解決）
2. 全キューブについて delay / dur / kind / scatter を計算し、現在姿勢→aQ0、目標→aQ1
3. 6属性の `needsUpdate = true`（このO(N)が唯一のコスト: 5Kで約4ms、50Kで約9ms）
4. spin 以外なら `current` 更新。`autoT = 0`（シネマ自動切替の抑制）
- **ベンチマークからは `rng` にシード付き乱数（mulberry32）を渡して決定的にする**

## 4. 座標系・不変条件（ここを壊すと絵が乱れる）

1. **面→画像の対応**: img0=+z, img1=-z, img2=+x, img3=-x, img4=+y, img5=-y。
   `FACE_QUATS[k]` は「面kを正面(+z)に正立で向ける」回転。順序を変えるなら全対応を同時に
2. **`aGrid` の行は反転して格納**: `grid[y] = ROWS-1-r`（r=0が最上段、テクスチャは v=0 が下端）。
   ワールド位置も同じ `aGrid` から `aGrid - (uGridSize-1)*0.5` で導出しているので、
   これを直すと位置とUVが同時に動く（意図的な設計）
3. **面ごとのUV補正は不要（identity が正解）**。BoxGeometry は全面のUVを
   「外側から見て正立」で張っており、回転で正面に来た面はまさに外側から見た状態。
   ⚠️ 過去バグ: `t.x`/`t.y` 反転を入れて**タイル内部だけが鏡像**になった。
   この症状は「画像全体は正しく、細部（数字・線画）だけギザギザ＋鏡像」という
   紛らわしい見え方をする。グリッドの継ぎ目と誤認しないこと（→ §7 検証手順）
4. カメラは常に +z 側。方位角ベースの処理（該当なし、Aurora Sea の教訓）を足すなら
   不連続点(±π)は -z ではなく +z 側に置く

## 5. パフォーマンス上の禁止事項・注意

- ❌ 毎フレームの `setMatrixAt` / instanceMatrix 更新に戻さない（〜5万で頭打ちになる旧方式）
- ❌ キューブを個別 Mesh 化しない
- テクスチャは最大6枚の sampler2D（if-chain 選択）。増やすならテクスチャ配列や
  アトラスを検討（WebGL2 前提なら `sampler2DArray` が素直）
- `renderer.setPixelRatio` は 1.75 でキャップ済み（Retina でのフィルレート対策）
- 200K時の頂点数は約480万/フレーム。頂点シェーダーに重い計算を足すときはベンチで前後比較
- 属性書き込み(O(N))はトリガー時のみ許容。毎フレームのO(N)ループを追加しない
- `preserveDrawingBuffer: true` はテストのピクセル検査用に必要（外さない）

## 6. テスト（E2E）の回し方

テスト本体はリポジトリに収録: **`tests/flip-wall.e2e.mjs`**

```sh
npm install                  # 初回のみ（playwright-core が入る）
npm run test:flip-wall       # リポジトリルートで実行
# Chromium の場所が違う環境では CHROMIUM_PATH=/path/to/chrome を指定
```

- 内蔵HTTPサーバーでリポジトリを配信し、ヘッドレスChromiumで実プレイ相当の検証を行う
- 36項目: 描画・輝度・アスペクト追従・6面切り替え・全パターン完走・シネマカメラ・
  自動切替・50K再構築・ミニベンチマーク・モバイルUI重なり検査・UI非表示 等
- スクリーンショットは `tests/output/`（.gitignore済み）に出るので**必ず目視も行う**（§7）

### ヘッドレス環境の落とし穴（実際にハマったもの）
- **低fps + dt クランプ(0.05)でゲーム内時間が壁時計より遅く進む**。
  「N秒待って判定」は必ずポーリングにする（固定 sleep はNG）
- GLドライバの性能警告（ReadPixels ストール等）を**シェーダーエラーと誤検知しない**。
  エラー判定は `THREE.WebGLProgram` / `ERROR:` 等に絞る
- Playwright の `page.screenshot()` は遅延があり**アニメ途中を撮り逃す**。
  途中経過は page 内で `canvas.toDataURL()` を setTimeout で撮って持ち出す
- ヘッドレスのFPS値はソフトウェアレンダリング（SwiftShader）なので絶対値に意味はない。
  「動く・完走する・相対比較」の確認に使う

## 7. 変更時チェックリスト

回転・ジオメトリ・シェーダー・グリッド周りを触ったら:

1. `tests/flip-wall.e2e.mjs` を実行して ALL PASS
2. **6面すべてをスクリーンショットで目視**（テストが出力する face 画像）。
   確認観点: 数字が正立か / 鏡像でないか / **輪郭がギザギザでないか**（タイル内鏡像の兆候）
   ※ アサーションは `current()==k` しか見ていない。向きの正しさは目視でしか担保されない
3. 集合パターンの途中フレームを目視（飛散→再集合が破綻していないか）
4. モバイルビューポート(390×844, isMobile)でUIの重なり検査＋目視
5. 触った箇所に応じて: シネマ各ショット / ベンチ完走 / 1枚モードのスピン動作

UI を触ったら: 下部UIは `#ui-bottom`（縦flex）の中に置く。**fixed の bottom 値で
バーを個別配置しない**（折り返しで重なった前科あり）。行は wrap させず横スクロール。
セーフエリア（`env(safe-area-inset-bottom)`）を維持。

## 8. デバッグフック（`window.__wall`）

| フック | 用途 |
|---|---|
| `grid()` | {cols, rows, n} |
| `current()` / `activeSet()` | 表示中スロット / 巡回対象 |
| `animating()` | アニメ中のキューブ数（ポーリングで完了待ち） |
| `fps()` / `triggerMs()` | 直近FPS / 直近トリガーの属性書き込みms |
| `trigger(target, pattern)` / `spin()` / `setPattern(p)` | パターン指定の切り替え |
| `switchTo(i)` / `clickCenter()` | UI相当の操作 |
| `loadDataURL(idx, url, cb)` | テスト画像の注入（ユーザー画像扱い） |
| `setCount(n)` | 粒子数変更＋再構築 |
| `benchmark(presets?, secsPer?)` | ベンチ実行（短縮版可）。Promise で結果を返す |
| `toggleCamera()` / `camMode()` / `camPos()` / `forceNextShot()` | カメラ制御 |
| `toggleUI()` / `uiHidden()` | UI表示切替 |
| `aspect()` | 壁のアスペクト |

## 9. 既知の改善候補（未着手・ユーザー未承認のものは実装前に提案する）

- 500K プリセット（シェーダーはそのまま動くはず。トリガーO(N)が~50ms級になる点だけ注意）
- `sampler2DArray` 化（if-chain 削減）/ 画像枚数の拡張
- IndexedDB による画像の永続化（現状はリロードで消える。プライバシー方針との整合を明記すること）
- シネマショットの追加・ショット間のブレンド（現状はハードカット）
- 集合パターンのバリエーション（渦巻き集合、文字形への集合など）
- モバイルの初期粒子数自動調整（実測FPSベースのオートスケール）
